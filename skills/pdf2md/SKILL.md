# SKILL: pdf2md

## Summary

将 PDF 忠实转换为 Markdown。

目标是**最大限度保留原文内容、顺序、层级、列表、链接、图示位置与页面信息**，而不是做总结、提炼、改写或结构优化。

适用于用户明确要求：

- 原汁原味
- 忠实还原
- 不要总结
- 不要改写
- 不要自行调整结构

---

## When to Use

在以下场景使用本 Skill：

1. 用户要求把 PDF 转成 Markdown
2. 用户明确要求"原汁原味""忠实还原""不要总结"
3. PDF 需要保留图文对应关系
4. 交付物需要是 `.md` 文件，必要时还要带图片资源包

---

## When Not to Use

以下场景不应直接使用本 Skill，或者应先澄清：

1. 用户要的是"摘要""提纲""导读""总结版"
2. 用户希望你"优化排版""重组结构""润色措辞"
3. 用户要的是 Word、PPT、Excel，而不是 Markdown
4. 用户只想提取某几页或某几个章节，且明确允许省略其他内容

---

## Success Criteria

只有当以下条件同时满足时，任务才算完成：

1. Markdown 与 PDF 的主体内容顺序一致
2. 没有总结、改写、重组
3. 标题层级基本对应原文
4. 列表、编号、段落没有被随意合并或拆分
5. 图示已保留，并在 Markdown 中以合适位置引用
6. 页脚中的原始链接、来源等重要信息未丢失
7. 产物可直接交付给用户使用

---

## Inputs

输入通常包括：

- 用户上传的 PDF
- 用户要求，例如：
  - "转成 markdown"
  - "原汁原味"
  - "不允许自行调整或总结"

---

## Outputs

标准输出：

1. `document.md`
2. 如有图片，输出资源目录：
   - `assets/...`
3. 如需便于交付，再输出：
   - `document_markdown_bundle.zip`

---

## Constraints

执行时必须遵守以下约束：

1. **不得总结**
2. **不得改写**
3. **不得自行优化结构**
4. **不得擅自删减边界内容**
5. **不得用文字摘要替代原图**
6. **不得在不确定时凭空补写**
7. **仅做必要的 Markdown 语法映射**

---

## High-Level Strategy

优先判断 PDF 是否存在可用文本层。

- 如果有可用文本层：优先走文本提取路线
- 如果没有可用文本层：按页读取渲染图逐页还原
- 如果是图文混排：同时处理文本与图片资源
- 如果页面中包含评论区、推荐阅读、页脚链接等边界内容：依据"原汁原味"原则谨慎保留，而不是主观裁剪

---

## Procedure

### Step 1: Inspect the PDF

先判断 PDF 的可解析性。

需要回答三个问题：

1. 是否存在可直接读取的正文文本层
2. 文本层是否完整、连续、可用
3. 页面中是否包含大量图片、截图、图示或整页截图内容

#### Decision

- 文本层完整：进入 Step 2A
- 文本层缺失或严重不完整：进入 Step 2B

---

### Step 2A: Extract from text layer

若 PDF 文本层可用：

1. 直接提取文本
2. 按页面/段落顺序整理
3. 恢复标题层级
4. 恢复列表与编号
5. 检查是否有未被文本层覆盖的图示、表格或页脚链接
6. 如有图示，补充图片导出与引用

注意：
即使文本层可用，也不能忽略页面中的图示和页脚信息。

---

### Step 2B: Reconstruct from rendered pages

若 PDF 文本层不可用：

1. 获取每一页的渲染图
2. 逐页阅读页面内容
3. 按自然阅读顺序还原：
   - 标题
   - 作者
   - 时间
   - 正文段落
   - 小节标题
   - 列表
   - 页脚链接
   - 页面内图示
4. 对小字、局部页脚、图下注释等内容，必要时局部放大或裁剪辅助识别

注意：
这是"逐页忠实还原"模式，不是"看懂后重新写一遍"。

---

### Step 3: Map content to Markdown

将识别出的内容映射为 Markdown。

#### 标题映射

- 一级标题 → `#`
- 二级标题 → `##`
- 三级标题 → `###`

#### 正文映射

- 按原顺序分段
- 不随意合并段落
- 不为了"更通顺"而改写

#### 列表映射

- 无序列表 → `-`
- 有序列表 → `1.` `2.` `3.`

#### 图片映射

- 将原图导出到 `assets/`
- 在对应位置插入图片引用

例如：

```md
![水平扩展的两种模型](assets/fig-01.png)
```

#### 链接映射

- 原始 URL 尽量保留
- 页脚来源链接必须保留
- 不擅自改成摘要文字

---

### Step 4: Preserve boundary content conservatively

当用户要求"原汁原味"时，边界内容默认应偏向保留。

通常包括：

- 标题
- 作者
- 发布时间/编辑时间
- 原文内链提示
- 页脚来源链接
- 图示与图下注释
- 页面截图中的评论区
- 页面截图中的推荐阅读区域

除非用户明确要求"只保留正文"，否则不要主观删掉这些内容。

---

### Step 5: Export images

若文中存在图示、结构图、流程图、插图或截图块：

1. 单独导出图片文件
2. 放入 `assets/`
3. 命名清晰稳定，例如：
   - `fig-01.png`
   - `fig-page3-arch.png`
4. 在 Markdown 原文对应位置插入引用

要求：

- 不省略图
- 不重绘图
- 不拿一段文字替代图
- 不擅自改动图意

---

### Step 6: Assemble final artifact

推荐产物结构：

```text
document.md
assets/
  fig-01.png
  fig-02.png
  fig-page3-arch.png
```

若要方便交付：

```text
document_markdown_bundle.zip
```

---

### Step 7: Validate fidelity

交付前必须检查以下内容。

#### Completeness

确认没有遗漏：

- 页面
- 标题
- 段落
- 列表
- 图示
- 页脚链接
- 页面中的原始附加内容

#### Order

确认：

- 段落顺序正确
- 标题顺序正确
- 列表顺序正确
- 图片位置没有明显错位

#### Fidelity

确认：

- 无总结
- 无改写
- 无结构重组
- 无擅自删减
- 无臆测性补写

#### Usability

确认：

- Markdown 语法可用
- 图片链接可访问
- ZIP 包完整
- 文件名清晰

## Failure Modes

以下是常见失败模式。

### 1. 把"忠实转写"做成了"内容整理"

表现：

- 自动补目录
- 重组章节
- 把原文压缩成更短版本
- 改写为"更流畅"的语言

这是错误的。

---

### 2. 忽略图示

表现：

- 只保留文字
- 图示未导出
- 图示位置丢失
- 用一句话概括图内容代替原图

这是错误的。

---

### 3. 错误处理边界内容

表现：

- 把评论区、页脚链接、推荐阅读全删了
- 或者反过来，把明显无关的 UI 噪音过度保留

处理原则应基于用户要求。
当用户要求"原汁原味"，优先保留，而不是主观精简。

---

### 4. 过度依赖 OCR

表现：

- 不先检查文本层
- 一上来就 OCR
- OCR 错字后不核对

正确做法是：

- 文本层优先
- 页面渲染图辅助
- OCR 仅在别无选择时使用

---

### 5. 段落与图片错位

表现：

- 图片插错位置
- 图后文字跑到前面
- 小节标题顺序错乱

解决方法：

- 始终按页建立阅读顺序
- 逐页复核图文相对位置

---

## Tests

执行完成后，至少做下面这些检查。

### Test 1: Page coverage test

核对 PDF 页数与输出内容覆盖范围是否一致。

检查点：

- 是否每页都处理到了
- 是否存在漏页

---

### Test 2: Heading consistency test

检查 Markdown 标题层级是否对应原文。

检查点：

- 一级标题是否唯一且正确
- 小节标题是否按原顺序出现
- 编号是否丢失

---

### Test 3: Image reference test

检查图示是否完整。

检查点：

- 每个导出的图片都被引用
- 引用路径正确
- 图片位置基本合理

---

### Test 4: No-summary test

检查输出中是否出现不属于原文的总结性文字。

例如以下内容不应凭空出现：

- "本文主要讲了……"
- "总结一下……"
- "核心观点是……"
- "简而言之……"

---

### Test 5: Link preservation test

检查是否保留原始链接和来源。

检查点：

- 页脚 URL 未丢失
- 原文中出现的链接未被擅自替换

---

## Implementation Notes

### Note 1: Text-first policy

优先使用 PDF 原生文本层。
只有当文本层不可用或不完整时，才退到页面渲染图路线。

### Note 2: OCR-last policy

OCR 是最后手段，不是默认方案。

### Note 3: Conservative preservation policy

对"原汁原味"任务，保留的成本低于删错的成本。
拿不准时，优先保留。

### Note 4: Visual order matters

这类任务不是"语义改写任务"，而是"文档重建任务"。
页面上的相对顺序很重要。

## Recommended Final Response Template

交付时，回复应简洁：

```text
已按原文忠实转为 Markdown，未做总结或改写。
附件包括：
1. Markdown 文件
2. 图片资源打包文件
```

不要在交付说明中加入额外内容分析。

## Example Invocation

用户输入：

```text
将这篇文档从pdf转为markdown。要求：原汁原味，不允许自行调整或总结
```

执行策略：

1. 检查 PDF 是否有文本层
2. 无文本层则按页读取渲染图
3. 逐页还原为 Markdown
4. 导出图示
5. 打包 Markdown + assets
6. 交付结果

---

## Minimal Checklist

执行前后快速检查：

- 是否判断了文本层是否可用
- 是否保留了标题层级
- 是否逐页处理了正文
- 是否保留了图示
- 是否保留了页脚链接
- 是否没有做总结和改写
- 是否输出了 `.md`
- 是否在有图时输出了资源包

---

## One-Line Rule

**把 PDF 当成需要"重建"的原文档，而不是需要"理解后重写"的内容。**
